## Context Pressure Tiers

The response to context pressure should be proportional to the severity. More aggressive compression at higher pressure, but always preserving the most recent and most critical information.

### Tier 1: Low Pressure (<70% full)

**Action**: None needed.
**Rationale**: Full fidelity maintained. All tool results, conversation history, and context preserved at original quality.

### Tier 2: Medium Pressure (70-85% full)

**Action**: Selective compression of older tool results.
**Strategy**: Identify tool results that are:
- Large (exceeding a size threshold)
- Re-obtainable (can be regenerated by re-running the same tool)
- Low information density (verbose output where key facts can be summarized)

**Preserve**: Recent messages, structured outputs, unique data that cannot be regenerated.

### Tier 3: High Pressure (85-95% full)

**Action**: Micro-compaction of older tool results.
**Strategy**: Summarize large outputs while preserving key data points. Focus compression on:
- File read results: Keep function signatures, class definitions, key logic — remove boilerplate
- Shell output: Keep relevant lines, remove noise
- Search results: Keep matches, remove surrounding context

**Preserve**: The last N messages at full fidelity (these drive current reasoning).

### Tier 4: Critical Pressure (>95% full)

**Action**: Full conversation compaction.
**Strategy**: Summarize the entire conversation history into a condensed context block. Preserve:
- The original task/request
- Key decisions made
- Files modified and changes made
- Current state and what remains to be done
- Any errors encountered and their resolutions

**Trade-off**: Maximum context recovery but some nuance lost in summarization.

## Selective Compression Principles

### What to Compress

1. **Large file reads**: A 500-line file read can be summarized to its key structures (classes, functions, imports) without losing the information needed for subsequent decisions
2. **Verbose shell output**: Build logs, test output with many passing tests — keep summaries and failures
3. **Broad search results**: When a search returns 50 matches, keep the relevant ones and a count of the rest
4. **Re-obtainable data**: Any information that can be regenerated by re-running the same tool call

### What to Never Compress

1. **The most recent messages**: These drive the agent's current reasoning — compressing them degrades decision quality
2. **Unique, hard-to-reproduce data**: User-provided context, one-time observations, specific error messages
3. **Structured outputs**: JSON, configuration files, data schemas lose meaning when summarized
4. **Agent directives and constraints**: The task definition and constraints must remain intact
5. **Partial results from sub-agents**: These represent work already done and cannot be easily reproduced

### Compression Decision Framework

```
For each tool result in context:
  Is it recent? → PRESERVE
  Is it small (< threshold)? → PRESERVE
  Is it re-obtainable? → COMPRESS (summarize)
  Is it high information density? → PRESERVE
  Is it low information density? → COMPRESS (summarize)
  Is it unique/hard-to-reproduce? → PRESERVE
```

## Sub-Agent Context Pruning

When spawning sub-agents, the parent's context is often too large to pass entirely. Prune strategically:

### What to Omit for Sub-Agents

1. **Project convention files** (README, style guides, contributing guides) — if the parent agent already has them and will interpret the sub-agent's output, the sub-agent doesn't need them too
2. **Irrelevant conversation history** — only pass the messages relevant to the sub-agent's specific task
3. **Other sub-agents' outputs** — prevent context pollution between sibling agents
4. **System-level environment details** — unless the sub-agent specifically needs them

### What to Always Include

1. **The sub-agent's specific directive** — its task, scope, and output format requirements
2. **Relevant file paths and references** — the sub-agent needs to know what to look at
3. **Constraints and limitations** — what the sub-agent must not do
4. **Output expectations** — format, length, level of detail

### Token Savings Math

If a parent context is 50K tokens and project conventions are 5K tokens, pruning them from each of 4 sub-agents saves 20K tokens total. In multi-agent systems with many sub-agents, these savings compound significantly.

### Context Budget Per Sub-Agent

A practical guideline:
- **Read-only sub-agents**: 10-20K tokens of context (they don't need full history)
- **Implementation sub-agents**: 20-40K tokens (they need more context to make correct changes)
- **Verification sub-agents**: 15-30K tokens (they need to understand what changed, not the full history)
- **Parallel workers**: Inherit parent context minus conventions and sibling outputs

## Prompt Cache Optimization

LLM providers offer prompt caching — when sequential requests share a common prefix, the cached portion is processed faster and cheaper. This is especially powerful for parallel sub-agent spawning.

### The Cache Sharing Principle

When spawning multiple parallel sub-tasks that inherit parent context:

```
Parent conversation history (identical across all children)
  + Parent's accumulated tool results (identical across all children)
  + Placeholder tool results for every tool_use block (identical across all children)
  + Child-specific directive (ONLY difference per child)
```

All children produce **byte-identical request prefixes**. Only the final text block differs. This maximizes cache hits.

### Requirements for Cache Sharing

1. **Same model**: Different models cannot share cache — use the same model for parent and all children
2. **Identical prefix order**: Messages must appear in the exact same order across all children
3. **Identical tool results**: Placeholder tool results must be byte-identical (not just semantically equivalent)
4. **Minimal suffix**: Only the child's directive should differ — don't add agent-specific metadata to the prefix

### What Breaks Cache Sharing

- Using different models for different children
- Adding agent-specific context to the prefix (e.g., "You are agent A")
- Changing the order of messages between children
- Using non-identical placeholder content

### When NOT to Use Context Inheritance

| Scenario | Better Approach | Why |
|----------|----------------|-----|
| Independent analysis of different files | Fresh start | Reviewer doesn't need parent's findings |
| Different domain investigation | Fresh start | No relevant context to inherit |
| Agent needs different model | Fresh start | Different models break cache sharing |
| Sub-agent needs minimal context | Fresh start with brief | Full inheritance wastes tokens |

## Context Inheritance Patterns

### Full Inheritance

Pass the parent's complete conversation history plus the child directive.

**Best for**: Sub-tasks that need the parent's full analysis to build upon.
**Cost**: Higher token usage per child, but maximized cache hits.

### Filtered Inheritance

Pass relevant portions of the parent's history plus the child directive.

**Best for**: Sub-tasks that only need specific context from the parent.
**Cost**: Lower token usage, but may miss relevant context.

### Summary Inheritance

Summarize the parent's findings, then pass the summary plus the child directive.

**Best for**: Sub-tasks where the parent's conclusions matter but the reasoning doesn't.
**Cost**: Lowest token usage, but loses nuance.

### No Inheritance (Fresh Start)

Pass only the child directive with relevant background.

**Best for**: Independent tasks, verification tasks, or when the parent's context is irrelevant.
**Cost**: Lowest per child, but no cache sharing benefit.

### Decision Framework

```
Does the child need the parent's specific findings?
  YES → Does the child need the full reasoning process?
    YES → Full inheritance (or filtered if very large)
    NO → Summary inheritance
  NO → Fresh start
```
